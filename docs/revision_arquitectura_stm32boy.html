Revisión de Organización y Abstracción del Proyecto STMBOY

Generado por Codex el 7 de marzo de 2026

Resumen

La base es buena (ya existe separación inicial source / bsp / platform), pero actualmente hay acoplamiento entre capas y mezcla de responsabilidades que dificulta escalar, portar a otras placas y mantener la API limpia.

Correcciones Detectadas (Prioridad Alta)

  1. Acoplamiento de main con la placa: source/main.c incluye y llama APIs específicas de nucleof411re. La capa de aplicación debería depender de BSP/API genérica, no de drivers de placa.
  2. Doble inicialización del display: El display se inicializa en main y también desde platform_init(). Debe existir un único punto de inicialización.
  3. Módulo con responsabilidades mezcladas: platform_nucleof411re_ili9486.c contiene pinout, bus paralelo, protocolo del controlador y funciones de dibujo. Conviene separar por capas (board/bus/driver/gfx).
  4. API pública inconsistente: Hay funciones declaradas pero no implementadas o duplicadas en headers. Esto genera deuda técnica y ruido para quien use la API.
  5. Assets dentro de header de API: El sprite sprite_alien_16x16 y macros de color están en stm32boy.h. Ese header debería exponer solo tipos y funciones públicas.
  6. Inconsistencia de resolución: Existen valores diferentes para ancho/alto (ej. 320x480 y 380x480). Debe definirse una única fuente de verdad para dimensiones de pantalla.
  7. Componente ILI9486 infrautilizado: El módulo genérico lib/platform/component/ili9486 casi no encapsula lógica. La mayor parte está todavía en la capa board.
  8. Documentación insuficiente: El README.md raíz es mínimo y no describe arquitectura, capas ni convenciones.

Recomendaciones de Organización (Estructura Objetivo)

  1. Separar en 4 capas: app (source), gfx (stm32boy), drivers (ili9486 genérico), board (nucleo f411re).
  2. Aislar la aplicación: main.c debe usar solo API de alto nivel (ej. bsp_init(), gfx_*), sin includes de placa.
  3. Mover stm32boy fuera de board: llevarlo a algo como lib/gfx/stm32boy/.
  4. Dejar en board solo lo específico de hardware: reloj, UART, pinmux, bus DB[7:0], timings y wiring.
  5. Mover assets/font a carpeta dedicada: por ejemplo lib/gfx/assets/ o assets/, evitando datos pesados en headers públicos.
  6. Convertir LUT a unidad de compilación: generar db_lut.c + db_lut.h declarativo para reducir recompilación y acoplamiento.
  7. Normalizar naming y API: una convención única (ej. snake_case), sin aliases duplicados.
  8. Documentar dependencias permitidas entre capas: crear docs/ARCHITECTURE.md con reglas claras.

Plan de Refactor Recomendado

Fase 1 (sin romper comportamiento):

  • Eliminar doble init y limpiar includes de main.
  • Depurar headers: quitar prototipos no implementados y duplicados.
  • Unificar constantes de resolución en una sola definición.

Fase 2 (reorganización por capas):

  • Mover stm32boy a lib/gfx/ y actualizar CMake.
  • Mover sprite/font/LUT a módulos dedicados.
  • Reducir platform_nucleof411re_ili9486.c a adaptación board/bus.

Fase 3 (mantenibilidad):

  • Mejorar README y añadir guía de arquitectura.
  • Añadir pruebas básicas de API (al menos compilación por preset).
